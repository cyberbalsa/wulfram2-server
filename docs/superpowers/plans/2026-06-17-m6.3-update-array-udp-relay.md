## Problem (live-confirmed with two rendering clients)
Players spawn and are relayed (engine entities, `entities=39`) but **clients don't see each
other in-game**, and motion is **not smooth / no deltas** (player's words: "they still need
deltas"). They DO see the server's demo tanks moving (jerky).

## Root cause (confirmed against the working Wulf-Forge reference)
The in-game entity relay must be **per-tick, per-session, over UDP**:
- **`0x0E` UPDATE_ARRAY** for **OTHERS** (every entity except the receiving session's own tank).
- **`0x0F` VIEW_UPDATE** for **SELF** (the receiving session's own tank + HUD stats).

Reference: `Wulf-Forge/main.py` `player_sim_tick` (~line 614) loops sessions and does
`udp_context.send(b'\x0E' + build_update_packet(others, seq, is_view_update=False, local_stats))`
and `... b'\x0F' + build_update_packet([self], seq, is_view_update=True, local_stats)`. Builder:
`Wulf-Forge/core/entity_manager.py` `build_update_packet` â†’ `network/packets/update_array.py`
`UpdateArrayPacket`.

**We instead send `0x0F`-for-everything over reliable TCP, once per visibility change (now per
tick after `cd6150d`), with no real timestamp and no self/others split.** So the client renders
entities it already knew from the initial `0x0F` snapshot (the demo tanks) but never receives the
`0x0E` "others" stream it uses to track in-game players or interpolate. â†’ invisible players + no deltas.

## Wire format (from `UpdateArrayPacket`, MSB-first bits â€” matches our `BitWriter`)
THE ONLY 0x0E-vs-0x0F DIFFERENCE is the leading timestamp:
```
0x0F (is_view_update=True):  [i32 timestamp = ms clock][i32 sequence] <stats> <count> <entities...>
0x0E (is_view_update=False):                           [i32 sequence] <stats> <count> <entities...>
```
- `<stats>`: `[bool has_stats]`; if true: `[5-bit pad/id = 0][COMPRESSOR_STAT health][COMPRESSOR_STAT energy]`.
- `<count>`: `[u8 entity_count]`.
- `<entity>` (`EntitySerializer.serialize`, strict order):
  `[i32 net_id][bool is_manned][10-bit mask][16-bit bank_selector = 0]`
  then if `mask & DEFINITION(bit0)`: `[unit_type:8][team:8][team:8][type-specific][bool force_snap=1]`
  (type-specific: unit_type 37 â†’ two i32 zeros; 19 â†’ cargo type 8 bits)
  then in strict bit order: `bit1 POS`, `bit2 VEL`, `bit3 ROT`, `bit4 SPIN` (each = vec3
  `[priority hdr = (1<<precision_header_bits)-1][Xc][Yc][Zc]`), `bit5 HEALTH` (COMPRESSOR_STAT),
  `bit6 WEAPON` (skip), `bit7 ENERGY` (COMPRESSOR_STAT), `bit8 OWNER` (i32 0), `bit9 HARD` (flag, no payload).
- Compressors (`Wulf-Forge/network/translation_config.py`): `COMPRESSOR_POS/VEL/ROT/SPIN/STAT`.
  Cross-check our `kPosConfig/kVelConfig/kRotConfig/kSpinConfig/kStatConfig` (world_packets.cpp ~423)
  match max/range/bits before trusting (kRotation max 6.3/range 12.6 already matches COMPRESSOR_ROT).

## What we already have (reuse)
- `BuildViewUpdateSnapshot(sequence, entities, health, energy)` (world_packets.cpp:609) = the
  `0x0F` shape but writes `WriteI32(sequence)` **twice** (should be `[ms timestamp][sequence]`).
- `WriteEntity` = the per-entity serializer for our mask subset (DEF|POS|ROT|HEALTH). The reference
  EntitySerializer is a superset (adds VEL/SPIN/ENERGY/OWNER); our subset is valid (client reads per mask).
- Quant machinery (`Quantize` clamps â€” no overflow), masks (`kMask*`), `Frame(opcode, writer)`.

## Implementation steps
1. **Parameterize the encoder.** Add
   `BuildUpdateArray(Opcode op, bool is_view_update, std::int32_t timestamp_ms, std::int32_t sequence,
   const std::vector<MvpEntitySnapshot>& entities, float health, float energy)`:
   if `is_view_update` write `WriteI32(timestamp_ms)`; always `WriteI32(sequence)`; then the shared
   stats+count+entities body; `Frame(op, writer)`. Refactor `BuildViewUpdateSnapshot` to call it
   (`op=ViewUpdate(0x0F)`, `is_view_update=true`). Add `Opcode::UpdateArray = 0x0E` (confirm it's in
   the opcode enum + IsKnownOpcode).
2. **Per-session emit** (`EmitSnapshots`): for each session with `wants_updates`:
   - `self_id = session.entity.net_id` (the engine oid from `SpawnPlayerTank`).
   - `others = [e for e in world if e.net_id != self_id]`; `self_vec = [the entity with net_id==self_id]`
     (from the engine world readback, or `session.entity`).
   - Send `0x0E`(others, is_view_update=false, seq) **via UDP**.
   - Send `0x0F`(self_vec, is_view_update=true, timestamp_ms=now, seq, session health/energy) **via UDP**.
3. **UDP transport.** Verify how to send a datagram to a session: the acceptor links a UDP endpoint
   per session (M5.2). Either extend `OutboundMessage` (reliable=false â†’ UDP) or add a dedicated
   per-session UDP send to the linked endpoint. THIS IS THE MAIN UNKNOWN â€” confirm the path first.
4. **Real timestamp.** Use a monotonic ms clock for the `0x0F` timestamp (client interpolates between
   updates using it â€” likely required for smoothness; `sequence` can stay the per-tick counter).
5. **DEFINITION bit.** Set `mask|=DEFINITION` for an entity the first time a given session sees it
   (and on (re)spawn), so the client creates/renders it; steady-state updates send POS|ROT (|HEALTH).
   Per-session "already-introduced" tracking, or force DEFINITION for the first N sends.
