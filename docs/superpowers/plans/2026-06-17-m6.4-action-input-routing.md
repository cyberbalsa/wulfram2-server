# M6.4 — ACTION input routing: drive a player's tank from their `0x09`/`0x0A`

**Status:** drafted 2026-06-17 (overnight). Wire-format section pending the in-flight Ghidra RE agent.
**Predecessor:** M6.3 (`3dc283c`) — the UDP `0x0E`/`0x0F` relay. With M6.3 a player's tank is spawned,
visible, and relayed, but the server's copy **never moves** when the player drives — only the 3 demo
tanks (`DriveServerTanks` from canned `kDemoDrive`) move. So via `0x0E` two players see each other
frozen at spawn. M6.4 closes that: decode the client's control inputs and drive the player's engine tank.

## Goal / done-when
A spawned player's tank moves **server-side** in response to that client's `ACTION` packets, the motion
is integrated by the engine's force-driven physics (same path as the demo tanks), and the resulting
authoritative position is relayed to peers via the existing `0x0E`. Done-when: two clients drive and
**see each other move where they actually drove** (not dead-reckoned divergence), no crash, no desync.

## What's already scaffolded (no new design needed)
- `ClientCommandKind::ActionInput` exists (`queues.hpp`) with `channel:i32` + `value:float` (value is
  meant to be clamped to `[-1,1]` at validation). Currently **no producer**, and the consumer is a
  no-op: `world_packets.cpp:753  case ClientCommandKind::ActionInput: return;`
- Inbound dispatch hook: `Connection::OnUdpPacket` (`connection.cpp:279`) — opcode branch ladder.
  `0x09`/`0x0A` are whitelisted (`validate.cpp:25`) but fall through to no-op.
- Apply surface: `DriveServerTanks` (`world_host_engine.cpp:407`) writes, per tank,
  `value*kControlDivisor(50)` to the shared tuning slots then calls `Vehicle_UpdateThrustSimulation`:
  - slot **1** = turn  (`kSlotTurn`)   → `slot_base + 1*0x20 + 0x10`
  - slot **2** = forward (`kSlotForward`) → `slot_base + 2*0x20 + 0x10`
  - slot **5** = thrust-intensity gate (`kSlotThrustIntensity`, write `1.0`) → `+0x18`
  - handler install (`entity+0xc0 = kForceDrivenHandlerVA`) + per-tick sleep-flag clear (`entity+0xae=0`).
- Player tanks: `SpawnPlayerTank` (`world_host_engine.cpp:244`) spawns the entity (oid `9100+`) but does
  **NOT** build a vehicle controller or register it in `DriveState().tanks` — so player tanks are visible
  but undriven. M6.4 must register + build controllers for them (mirror `SetupServerTanks`).

## Wire format — `ACTION_UPDATE 0x0A` / `ACTION_DUMP 0x09`  *(VERIFIED — Ghidra + 826 captures)*
From 826 golden captures (`tools/captures/action_packets_2026-06-17.txt`), raw UDP `[opcode][body]`:
```
0a 01 00 02 0b d7 00 00 00 00 00 08 00      (13B)
0a 01 00 00 50 83 00 00 00 00 00 06 85 87   (14B)
```
Working hypothesis (to be replaced by the RE agent's verified spec):
- `[0]` opcode (`0x0a`/`0x09`)
- `[1]` count of actions (`0x01` in all single captures)
- `[2..5]` i32 **timestamp** (big-endian-looking; the field that increments across packets)
- `[6..9]` i32 **entityId** (`0` here = local self)
- `[10..]` bit-packed per action: **channel index** bits + **quantized value** bits.
  Channel-index numbering and per-channel value encoding (bit width / quant range / signedness) come
  from `Net_SendActionUpdate@0x46C860` + its channel-value writer. The 196× `…00 00 00 00` frames are
  zero/release. **Must map ACTION channel index → the tuning slot it drives** (forward/turn/strafe/
  jumpjet/fire). The decoder output is `(channel, value∈[-1,1])` pairs — exactly `ActionInput`.

`0x09` (DUMP) = full channel set; `0x0A` (UPDATE) = changed channels only. Decode identically; DUMP just
carries more entries.

## Implementation (clean increments, each gated)

### Increment 1 — pure decoder (self-verifiable, TDD against real captures) ⟵ ship first
New `proto`/`server` free function `DecodeActionPacket(opcode, body, len) -> {timestamp, entityId,
vector<ActionChannelValue{channel,value}>}` using the existing `BitReader`. No engine, no threads.
- **TDD:** feed the literal captured byte arrays above; assert the decoded channel+value matches the
  hand-decode from the RE spec. Assert malformed/short bodies reject cleanly (no OOB read).
- This is the risk-bearing wire logic and it is **fully testable without a client** — land it alone.

### Increment 2 — producer wiring (net thread)
`OnUdpPacket`: when `state_==InGame` and opcode ∈ {`0x09`,`0x0A`}, call `DecodeActionPacket`, clamp each
value to `[-1,1]`, and push one `ClientCommand{ActionInput, session_id, channel, value}` per entry. Never
trust the client's `entityId` — the server maps `session_id → owned oid`. **TDD:** `Connection` test that
an in-game action datagram enqueues the expected `ActionInput` commands and a pre-game one does not.

### Increment 3 — consumer + apply (tick thread)
- `SessionState` gains a per-channel input cache (e.g. `std::array<float, kNumChannels>`), updated by
  `HandleCommand(ActionInput)`.
- Engine glue: when a session is spawned, register its oid in the drive system and build a controller
  (mirror `SetupServerTanks`). Map `session → oid → ServerTank`.
- `DriveServerTanks`: for player tanks, write the session's cached forward/turn (and strafe/jump once
  mapped) instead of `kDemoDrive`. Demo tanks keep their canned drive (or are disabled behind a flag for
  clean player tests).
- **TDD where pure:** the channel→slot mapping and the session→input cache update. The engine poke
  (controller build + slot write) is validated live (headless: server boots, drives a scripted action,
  readback shows the player oid moving), not in unit tests.

## Risks / unknowns
- **Bit format precision** — the whole decoder hinges on the exact channel-index width + per-channel
  quant. Mitigated by RE agent + golden-capture TDD. Until confirmed, do not ship the decoder.
- **Channel→slot coverage** — forward/turn are known slots; strafe/jumpjet/fire slots may need more RE.
  Increment 1/2 are slot-agnostic; only Increment 3 needs the mapping. Ship forward+turn first.
- **Threading** — input crosses the net→tick boundary via the existing queue (POD only, by value); no
  engine pointer crosses. Consistent with the established topology.
- **Controller-per-player lifecycle** — demo tanks build controllers once in `SetupServerTanks`; player
  tanks spawn/despawn dynamically. Need build-on-spawn + (later) teardown-on-despawn. Increment 3 detail.
- **Hover-while-driving** — a known open refinement (drive recipe note); tank may sink/climb under drive.
  Out of scope for M6.4; track separately.
