## Root cause (UNIFIED — the two "zero fields" are one bug)

A non-local tank doesn't move because the per-frame **thrust step never runs for it**. The stock
engine dispatches `Vehicle_UpdateThrustSimulation @0x4f9e20` (reached as **Tank_Model vtable[0]**,
vtable `0x5430e8`) for **exactly one object**: the singleton local-vehicle client `DAT_006782b0`,
and only inside `Interp_StepSimulationSubsteps @0x469e00` under the guard
`(DAT_006785e4 world != 0 && DAT_005b8392 == 0 && DAT_00677f2c local-entity != 0)`. A headless
`Obj_Create`/`Obj_InitFromSpawn` entity sets none of those globals, so the step is never invoked.
Consequences (both symptoms of that one cause):

1. **`model+0x58/+0x5c` stay 0.** Their SOLE writer is `Vehicle_UpdateEngineGlowScale @0x4f9790`,
   which re-initializes them to `1.0f` (`0x3f800000`) **every frame** — but only when the thrust
   step runs. `VehicleModel_ConstructDefault @0x4e9590` writes only `+0x00/+0x04/+0x08`, so on a
   fresh `operator_new(0x80)` model they remain garbage/0. They are NOT config/file fields (the old
   "thrust magnitude loaded from config" label is a CONFIRMED MISNOMER).
2. **The control slot table is empty.** `*(DAT_0067cd58)` (base) is a 22×`0x20` slot array filled
   per-tick by `LagQueue_AccumulateAndFlushDue @0x46aca0` from LOCAL input only (same local-gated
   branch). `DAT_0067cd58` is a SINGLE SHARED GLOBAL (sole writer `Interp_CreateManager @0x46ab44`),
   DISTINCT from the BEHAVIOR-loaded per-model registry `DAT_005c41c8` — so the already-wired
   `FeedBehaviorToEngine()` loads per-model config doubles but cannot/does not touch the live slots.

In `Vehicle_ApplyThrustForces @0x4f9b10`: `+0x58` multiplies the forward & strafe terms, `+0x5c`
the turn & strafe terms — if either is 0, those forces are 0.

**The fix:** drive the authentic step ourselves per non-local entity each tick (with that entity's
own slot table populated and the two gates passed), then let the engine's global
`EntityPhysics_RunWorldTick @0x4f8550` integrate `entity+0x24 → entity+0x18 (vel) → entity+0x0c (pos)`
for free. RunWorldTick **zeros** `entity+0x24/+0x28/+0x2c` and `entity+0x48/+0x4c/+0x50` after each
integrate, so the slots/step must be re-applied EVERY tick.

## Verified field map (ground truth for the pokes)

- Thrust step `Vehicle_UpdateThrustSimulation @0x4f9e20` (`__thiscall` ECX=Tank_Model, stack=Tank_Vehicle):
  sets `model+0x48 = tankVehicle`, `model+0x4c = *(tankVehicle+4) = entity`. Gate1: `entity+0xd0`
  (health) != 0. → `ComputeControlScalars` → `UpdateThrustFx` (sets gate2 `TankVehicle+0x18` from
  tuning slot 5). Gate2: `TankVehicle+0x18` != 0. → `UpdateEngineGlowScale` (sets `model+0x58/+0x5c=1.0`)
  → `ApplyThrustForces`.
- Tuning table reached as `*(model+0x48)+8` = `*(TankVehicle+0x08)` (= `DAT_0067cd58` on local path);
  divisor `*(model+0x48)+0xc` = `TankVehicle+0x0c` = **int 50**. base = `*(table)`; slot n = base+n*0x20.
- **slot+0x10** = control field that `GetScaledValue` divides by divisor (write control inputs here;
  raw = input*50). **slot+0x18** = raw/current cache (gate slot 5 read via `GetValue`; write the
  thrust-intensity gate at slot5 = base+0xB8 = 5*0x20+0x18).
- `ComputeControlScalars @0x4f9540`: slot1 → NEG → `model+0x74` (TURN); slot2 → `model+0x70` (FORWARD);
  slot3 → NEG → `model+0x78` (STRAFE); each clamped to `[_DAT_00564fc0, 1.0]`.
- `ApplyThrustForces @0x4f9b10` accumulates LINEAR force onto **entity**+0x24/+0x28/+0x2c and TORQUE
  onto **entity**+0x48/+0x4c/+0x50 (entity = `*(model+0x4c)`). Integrated + zeroed by RunWorldTick.
- Per-entity table: `VehicleControlHistory_Construct @0x4f6610` (container `operator_new(0x10)` whose
  `+0x00` = `operator_new(0x2c0)` = 22×0x20 slots via `VehicleControlSample_Clear`).
- Slot writers to mirror: `VehicleTuningTable_SetScaleLimit @0x4f64e0` (slot+0x10),
  `VehicleTuningTable_SetValue @0x4f64c0` (slot+0x18). (We can do raw memcpy writes instead.)
- ACTION wire (NO engine receiver — client only SENDS): class selector `(7<id)+1`. id>=8||id==4 →
  1-bit bool; id==5 → Range descriptor `DAT_00678134+0x280`; ids 1,2,3,6,7 → Range descriptor
  `DAT_00678134+0x2c0` (high=1.0/span=2.0 → `[-1,1]`); channel-id width `DAT_00678134+0x3c0`.
  Dequant via `Range_LevelToValue @0x4f8f40`. Channel→slot: 1→turn, 2→forward, 3→strafe, 5→intensity.

## ACTION input decode spec — CONFIRMED against the Wulf-Forge reference (2026-06-17)

Cross-checked `../Wulf-Forge/main.py` `parse_action_packet` / `_read_action_value` (which cite the
engine encoders `0x46C790`/`0x46C860`/`0x4DDC60`) against our own engine decompile — they AGREE. The
exact wire format the server must decode (client→server only; there is NO engine receiver to call):
- **ACTION_DUMP (0x09):** `[u8 op][i32 dumpTimeOrSeq][i32 flags]` then an implicit full table for ids
  `1..21`, each value via the rule below.
- **ACTION_UPDATE (0x0A):** `[u8 op][u8 count][i32 timeOrSeq][i32 flags]` then `count`× pairs of
  `[id : cfg15.precision_header_bits][value]`.
- **per-value rule** (`_read_action_value`): `id>=8 || id==4` → read **1 bit** (→ 1.0/0.0);
  `id==5` → `read_quantized_float(cfg 10)`; ids `1,2,3,6,7` → `read_quantized_float(cfg 11)`.
- **channels:** 1=Turn, 2=Forward, 3=Strafe, 4=JumpJet, 5=Hover(Up/Down), 6=Tilting.
- **dequant:** mirror `Range_LevelToValue @0x4f8f40`. The cfg 10/11/15 params (precision bits, max,
  range) live in `../Wulf-Forge/network/translation_config.py` (`get_config_by_index`) and are the
  SAME values our own `BuildTranslation` sends the client — so read them from our C++ side (or that
  module) rather than guessing; cfg 11 is the [-1,1] turn/fwd/strafe channel.
- **channel → tuning slot (already verified):** 1→slot1 (turn, neg → model+0x74), 2→slot2 (fwd →
  model+0x70), 3→slot3 (strafe, neg → model+0x78), 5→slot5 (hover/thrust-intensity gate, base+0xB8).
