# M0–M2 Integration Smoke Test (inject + pin + self-check)

Date: 2026-06-16
Branch: `master`
Result: **PASS** — loader pins + injects, DLL `InitThread` runs and logs its markers
in the real `wulfram2.exe`, wrong-binary is rejected before any spawn, and no game
process is left running.

## What this verifies

The Milestone 0–2 foundation, end-to-end against the **real** game binary:

1. `loader.exe` validates the target against the pinned manifest (PE
   TimeDateStamp / SizeOfImage / CheckSum / ImageBase vs `kBinaryManifest`)
   **before** doing anything.
2. It creates the target suspended, injects `wulf_headless.dll`, and resumes.
3. The injected DLL's `DllMain` spawns `InitThread`, which inits logging, runs the
   hook-site self-check (0 sites at this milestone → trivially passes), and logs
   "foundation ready".

No head-chop / hooking exists yet (deferred to Milestone 3), so the game's own
video/audio/window startup after resume is out of scope here — the smoke test only
confirms OUR injection mechanics.

## Setup

```powershell
.\build.ps1 -SkipTests
New-Item -ItemType Directory -Force build\config | Out-Null
Copy-Item config\headless.toml build\config\headless.toml -Force
```

`wulf_headless.dll` was freshly rebuilt by the build; `config\headless.toml`
(`[log] level = "debug"`) was copied to `build\config\` so `InitThread` finds it.

## Positive smoke (real binary)

```powershell
Remove-Item logs\loader.log -ErrorAction SilentlyContinue
Remove-Item build\logs\headless.log -ErrorAction SilentlyContinue

# try/finally so the spawned game is always killed
try {
    & .\build\loader.exe .\w2\wulfram2.exe -windowed   # exit code 0
    # poll up to ~12s for build\logs\headless.log to contain the InitThread markers
}
finally {
    Stop-Process -Name wulfram2 -Force -ErrorAction SilentlyContinue
}
```

Loader returned **exit code 0** and printed `headless server started, pid=8964`.
The `headless.log` markers appeared effectively immediately (< 0.5 s of polling).

### `logs\loader.log`

```
[INFO][loader][t0][injector.cpp:83] binary pinning OK (stamp=45254d1f size=00288000)
[INFO][loader][t0][injector.cpp:207] created suspended pid=8964
[INFO][loader][t0][injector.cpp:161] injected wulf_headless.dll
```

### `build\logs\headless.log`

```
[INFO][init][t0][init.cpp:68] InitThread alive; logging up (level=1)
[INFO][init][t0][init.cpp:76] hook-site self-check OK (0 sites)
[INFO][init][t0][init.cpp:78] foundation ready; head-chop deferred to Milestone 3
```

All required markers present:

- loader: "binary pinning OK", "created suspended pid=", "injected wulf_headless.dll" ✔
- headless (in the **real** process, pid 8964): "InitThread alive",
  "hook-site self-check OK (0 sites)", "foundation ready; head-chop deferred" ✔

## Negative smoke (wrong binary rejected)

```powershell
.\build\loader.exe .\build\loader.exe      # point loader at itself: a real x86 PE
                                            # whose stamps differ from wulfram2.exe
```

- Exit code: **1** (non-zero) ✔
- stderr: `inject failed: PE TimeDateStamp mismatch (wrong wulfram2.exe build)` ✔
- `logs\loader.log` (appended): `[FATAL][loader][t0][injector.cpp:79] binary
  pinning failed: PE TimeDateStamp mismatch (wrong wulfram2.exe build)` ✔
- **No process spawned**: `wulfram2` count = 0 and `loader` child count = 0 before
  and after the run. Crucially, there is **no** `created suspended pid=` line after
  the FATAL entry — the pinning gate blocked execution before `CreateProcess`. ✔

## Cleanup confirmation

```powershell
Stop-Process -Name wulfram2 -Force -ErrorAction SilentlyContinue
Get-Process wulfram2 -ErrorAction SilentlyContinue   # -> none
```

- `wulfram2` processes after cleanup: **0**
- `loader` processes: **0**

No game process or window left running.

## Caveats

- The game's own post-resume startup (video/audio/window via DirectDraw/Glide/3dfx)
  is **not** exercised or asserted by this milestone — head stubbing is deferred to
  Milestone 3. In this run the injected `InitThread` logged its markers immediately,
  so the markers were captured well within the 12 s window regardless of whatever the
  game's video init did afterward. The spawned process was force-killed during cleanup.
- The hook-site self-check reporting "0 sites" is expected at this milestone; real
  hook-site byte capture is deferred to Milestone 3.
- `build.ps1` emits a benign `'vswhere.exe' is not recognized` line from the vcvars
  bootstrap; the build still configures and links successfully.
